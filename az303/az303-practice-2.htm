So, how does Azure Bastion meet the "shared external IP address" requirement mentioned in the question?

Let's take a quick look at the Azure Bastion architecture:

After you are connected to Azure portal, the first thing you do is connect to Azure Bastion. This connection is secured and encrypted through TLS. Azure Bastion is actually a host deployed on the same virtual network as your VMs, that you intend to connect to.

The virtual machines don't need to have public IP address attached so that you connect to them, only one public IP address is needed on the Azure Bastion.

So, once you connect to Azure Bastion IP's address - which represents the shared external IP address - you can then connect through RDP to the individual VMs. It is a shared external IP address, as this IP address can be used by multiple users to connect to VMs, at the same time.

Explanation

Azure Service Bus is a fully managed enterprise message broker with message queues and publish-subscribe topics. Data is transferred between different applications and services using messages .

Messages are sent to and received from queues . Queues store messages until the receiving application is available to receive and process them.

Queues offer First In, First Out (FIFO) message delivery to one or more competing consumers. That is, receivers typically receive and process messages in the order in which they were added to the queue . For this reason, taking a look at the order messages are sent and added to Queue1 (M3,M2,M1,M3), Client2 will receive the messages in the same order: M3,M2,M1, M3.

If we now take a look at the answer options, we see that only the last option meets the FIFO (First In First Out) Queue1 behavior.

There is one more setting applied when defining Queue1, which may seem confusing - Enable duplicate detection :

Enabling duplicate detection configures your queue to keep a history of all messages sent to the queue for a configurable amount of time. In this scenario, we see that the default 10 minutes interval has been configured:

During that interval, Queue1 will not accept any duplicate messages. Enabling this property guarantees exactly-once delivery over a user-defined span of time.

So Queue1 will receive four messages in the first place, in the following order - M3, M2, M1, M3. Once these four messages are received by Queue1, for the next 10 minutes - Duplicate detection window interval - Queue1 will not accept anymore the M3 message, as it has been received already (in the first sequence M3,M2,M1,M3) and duplicate detection is now activated. Duplicate detection only looks at MessageID duplication, ignoring any other difference in the rest of the message.

Question 8: Incorrect

You are designing an Azure solution. The solution must meet the following requirements:

- Distribute traffic to different pools of dedicated virtual machines (VMs) based on rules.

- Provide SSL offloading capabilities.

You need to recommend a solution to distribute network traffic.

Which technology should you recommend?

Explanation

Based on question requirements, you need to design a solution to distribute traffic to different pools of dedicated virtual machines (VMs) based on rules, so to different VMSSs (virtual machine scale sets).

For such a solution, we can use either an Azure Load Balancer or an Azure Application Gateway.

The second requirement that we need to take into consideration is that the proposed solution needs to provide SSL offloading capabilities and only Azure Application Gateway can perform this.

HTTP traffic nowadays is encrypted (HTTPS) between source and destination, so between the client and the server - part of the virtual machine scale set. For such scenarios, the web server needs to first decrypt the traffic and then process it, but the encryption process is very intensive and requires the web servers to have a powerful configuration (CPU, RAM) so that both encryption and regular traffic processing is performed.

For this reason, the Azure Application Gateway can perform traffic encryption and decryption with the client, and pass the traffic to the web servers in clear format, unencrypted. This process and capability is also known as SSL offloading.

Reference:

https://docs.microsoft.com/en-us/azure/application-gateway/features

Question 13: Correct

You have an app named App1 that uses data from two on-premises Microsoft SQL Server databases named DB1 and DB2.

You plan to move DB1 and DB2 to Azure. You need to implement Azure services to host DB1 and DB2. The solution must support server-side transactions across DB1 and DB2.

Solution: You deploy DB1 and DB2 to SQL Server on an Azure virtual machine.

Does this meet the goal?

Explanation

Microsoft Distributed Transaction Coordinator (known as MSDTC) appears in SQL Server 2014, And current functionality is available from SQL Server 2016 SP2.

MSDTC is a coordinator to handle distributed transactions. A distributed transaction involves two or more databases in a single transaction.

Imagine your application started a transaction that inserts records into two different databases db1 and db2. These databases exist on the different SQL Servers SQLNode1 and SQLNode2, or on the same SQL Server. Your transaction should succeed if it inserts records correctly in both databases involved in the transaction. In this case, MSDTC monitors the distributed transaction, and if any of the server transaction fails, it takes decisions to roll back the whole transaction.

Suppose once the distributed transaction starts, it inserted a record in the DB1, but before it inserts into the DB2, DB2 shuts down. This transaction is known as an in-doubt transaction. MSDTC’s role is to ensure that the in-doubt transactions are either rolled back to committed.


In this scenario, the proposed solution it to deploy both DB1 and DB2 on the same SQL server, running on an Azure VM, so it meets the goal.


Reference:

https://docs.microsoft.com/en-us/sql/t-sql/language-elements/begin-distributed-transaction-transact-sql?view=sql-server-ver15

Explanation

Before even taking a look at the answer options provided, let's first understand a bit more about the components and technologies includes in this scenario.

A failover group is a named group of databases managed by a single server (or within a managed instance) that can fail over to another region in case all or some primary databases become unavailable due to an outage in the primary region.

The Primary server is the server that hosts the primary databases in the failover group.

The Secondary server is the server that hosts the secondary databases in the failover group. Please note that the secondary server cannot be in the same region as the primary server , as no failover to secondary region would be possible in this case.

Failover groups can be populated by either adding single databases to the failover group or by adding databases in elastic pools to the failover group. Few notes on this process:

⦿ You can put several single databases on the same server into the same failover group. If you add a single database to the failover group, it automatically creates a secondary database using the same edition and compute size on secondary server.

⦿ You can put all or several databases within an elastic pool into the same failover group. If the primary database is in an elastic pool, the secondary is automatically created in the elastic pool with the same name (secondary pool).

Now, let take a look at the three statements:

Statement 1: You can add DB1 to Failover1 - True; DB1 is deployed on SQLServer1 and is not part of any pool yet, so it can be added to Failover1 failover group.

Statement 2: You can add DB3 to Failover1 - False; DB3 can't be added to the failover group since it is already part of the secondary server and has the same pool name - Pool1, to provide failover from the primary. DB3 should be deployed on SQLServer1, and when added to the failover group, the secondary DB will automatically be created in the elastic pool with the same name, in the secondary region.

Statement 3: SQLserver1 and SQLserver2 are in the same Azure region - False; Primary and Secondary servers need to be deployed in different Azure regions, for the failover group setup.
