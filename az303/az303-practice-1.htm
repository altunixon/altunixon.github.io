Question 1: Correct

You plan to create an Azure Storage account in West Europe Azure region. You need to create a storage account that meets the following requirements:

- Replicates synchronously

- Remains available if a single data center in the region fails

How should you configure the storage account? (Please select two)

Explanation

First, let's clarify what replication options exist in Azure. While creating an Azure storage account, you can choose the replication option that best meets your technical needs.

Azure Storage replication options:

- Locally Redundant Storage (LRS) - copies your data synchronously three times ( 3X ) within a single physical location in the primary region.

- Zone-redundant storage (ZRS) copies your data synchronously across three ( 3X ) Azure availability zones in the primary region

- Geo-redundant storage (GRS) copies your data synchronously three times ( 3X ) within a single physical location in the primary region using LRS. It then copies your data asynchronously to a single physical location in the secondary region. Within the secondary region, your data is copied synchronously three times ( 3X ) using LRS. - 6 total copies of your data

- Geo-zone-redundant storage (GZRS) copies your data synchronously across three Azure availability zones in the primary region using ZRS ( 3X ). It then copies your data asynchronously to a single physical location in the secondary region. Within the secondary region, your data is copied synchronously three times using LRS ( 3X ). - 6 total copies of your data

We need to choose a storage account type, based on restrictions presented in the question, as follows:

- Replicates synchronously

- Remains available if a single data center in the region fails

The second restriction means that we need to choose a replication type that will copy the data to multiple data centers (or Azure availability zones) in the same region. For this reason, LRS, GRS and RA GRS are not valid options, because data is copied in the primary region in the same Azure data center or physical location and if the data center fails, then data will not be available. So this clarifies the first option, we need to deploy a storage account and select Zone-redundant storage (ZRS) replication.

Last, we need to select one of the storage account types provided: Blob storage, Storage (general purpose v1) and StorageV2 (general purpose v2). The correct option is StorageV2 (general purpose v2) , because it's the only one that supports Zone-redundant storage (ZRS) replication, from the three options the question provides:

You have an Azure subscription that contains multiple resource groups. You create an availability set as shown in the following exhibit:

During an Azure planned maintenance, at least .......... virtual machined will be available.

Explanation

An availability set is a logical grouping of Azure virtual machines (VMs) that allows Azure to understand how your application is built, in order to provide redundancy and availability.

Each virtual machine in an availability set is assigned an update domain and a fault domain by the underlying Azure platform. By default, within an availability set, 5 update domains are assigned  to indicate groups of virtual machines and underlying physical hardware that can be rebooted at the same time. For this specific scenario, we see that the availability set has been configured with 3 update domains :

So, how do update domains help? What's the main benefit of update domains? The main point is that update domains can help you during Azure planned maintenance events, when either software updates are performed or hardware is changed or updated. Because Microsoft will never reboot all update domains at once, having your virtual machines spread on multiple update domains means that your application will remain online, being served by the resources that haven't been rebooted yet.

How does Azure spread your VMs on the configured Azure update domains? Let's take our scenario as an example. We have 10 virtual machines deployed in our availability set - AS1: VM1, VM2, VM3, VM4, VM5, VM6, VM7, VM8, VM9, VM10.

Because we have 3 update domains configured, the 4th VM will be placed in the same update domain as VM1, the 5th VM will be deployed in the same update domain as VM2, the 6th VM will be placed in the same update domain as VM3, and so on.

The end result is the following:

- Update Domain #1 : VM1, VM4, VM7, VM10 => 4 VMs total

- Update Domain #2 : VM2, VM5, VM8 => 3 VMs total

- Update Domain #3 : VM3, VM6, VM9 => 3 VMs total

Please note that only one update domain is rebooted at a time! A rebooted update domain is given 30 minutes to recover before maintenance is initiated on a different update domain.

So now, finally, let's take a look at the question presented in this scenario:

During an Azure planned maintenance, at least .......... virtual machined will be available.

Because the question is asking for the minimum virtual machines that can be available during a planned maintenance, in case Update Domain #1 is rebooted, there will be 6 VMs available (Update Domain 2 + Update Domain 3).

If the question would ask for the maximum VMs available, then the answer would be 7 VMs. This would happen if either UD #2 or UD #3 is rebooted.

You have a virtual network named vNET1, configured as shown in the following exhibit:

No devices are connected to vNET1. You plan to peer vNET1 to another virtual network named vNET2. vNET2 has an address space of 10.1.0.0/16. You need to create the peering.

What should you do first?

Explanation

Virtual network peering enables you to connect two or more Virtual Networks in Azure.

Once the peering connection is configured and running, the traffic between virtual machines in peered virtual networks uses the Microsoft backbone infrastructure. Similar to traffic between virtual machines in the same network, traffic is routed through Microsoft's private network only.

By default, VMs in the same vNET can communicate to each other. Once the vNET peering is configured, VMs from peered vNETs can communicate to each other, just as if they were deployed in the same vNET.

One of the limitations that you need to be aware of when considering connecting two or more vNETs through vNET peerings is that virtual networks must have non-overlapping IP address spaces . If virtual network address spaces overlap, they cannot be peered.

And it makes a lot of sense to be this way. Just think, if you have two VMs with the same IP address, 10.1.0.15 for example, deployed in two vNETs that you connect through a vNET peering, what happens after peering is up and running?

If another VM needs to send traffic to destination IP address 10.1.0.15 ... where will the traffic be sent? To which VM? To VM in vNET1 or to the VM in vNET2?

So the rule to remember is that the virtual networks you peer must have non-overlapping IP address spaces.

In this specific scenario, in order to peer vNET1 with vNET2, you would first need to modify the address space of vNET1 .

Note that in the vNET1 image, we can see there is not any device connected. But if we have some subnet using any part of the current address space of vNET1(We have not this information available in the image), the way we have to proceed is the following:

1.- Delete the subnets in vNET1 (if any)
2.- Add a new Address space to vNET1 compatible with vNET2
3.- Delete the 10.1.0.0/16
4.- Create again the subnets in vNET1 (if any)
5.- Create the peering between vNET1 and vNET2 in both senses (From the portal is done automatically, but using az commands or PorwerShell cmdlets, we need to create in both sides)
