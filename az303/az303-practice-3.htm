Question 1: Incorrect

You have virtual machines (VMs) that run a mission-critical application. You need to ensure that the VMs never experience down time.

You need to match the appropriate solutions to the correct scenarios, as presented below:

Scenarios:

1. Maintain application performance across identical VMs.

2. Maintain application availability when an Azure datacenter fails.

3. Maintain application performance across different VMs.

Solutions:

A. Fault Domain

B. Availability Zone

C. Availability Set

D. Scale Set

Explanation

There are multiple availability options that you can implement for Azure virtual machines (VMs), such as availability zones, availability sets and virtual machine scale sets.

Azure virtual machine scale sets (VMSS) allow you to create and manage a group of identical load balanced VMs. The number of VM instances part of the VMSS can automatically increase or decrease in response to traffic demand changes or a defined schedule. Since the first scenario is asking for maintaining application performance across identical VMs , virtual machine scale sets or simply scale sets are the solution in this case.

An Availability Zone is a physically separate zone, or data center, within an Azure region. Not all Azure regions support Availability Zones, so this should be checked in advance, when needed. When Availability Zones are supported, you should know that there are three Availability Zones per supported Azure region.

Availability zones are a great choice when you need to prevent application downtime because of Azure data center failures. Deploying your application in multiple data centers or Azure Availability Zones can help in this case, because if one data center fails, then your application will be kept online by the servers running in the other two Azure Availability Zones. Thus, Availability Zones help maintain application availability when an Azure datacenter fails.

An Availability Set is a logical grouping of VMs that allows Azure to understand how your application is built to provide for redundancy and availability. Each virtual machine in your availability set is assigned an update domain and a fault domain by Azure.

Update domains can help you during Azure planned maintenance events, because virtual machines deployed in different update domains are not rebooted at the same time. A rebooted update domain is given 30 minutes to recover before maintenance is initiated on a different update domain.

On the other hand, fault domains define the group of virtual machines that share a common power source and network switch. By placing your virtual machines into multiple fault domains, the impact of potential physical hardware failures, network outages, or power interruptions is greatly limited. Inside an Azure data center, if one rack of servers goes down - which you can think of a fault domain, your application will still be online, being served by the virtual machines deployed in other racks of servers, of fault domains. Thus, Azure availability sets can help maintain application performance across different VMs.

Incorrect Solution Answer:

Fault Domain is not a valid solution for the third scenario, because you can't configure a fault domain. Fault domains along with Update domains are part of Azure Availability Sets, which represent one configurable option. Once you select Availability Sets during VM deployment, you can then decide how many fault domains and update domains you need and want to configure.

③ Use VM scale sets.

Bottom line, the correct solution for this scenario (and production environments) is to use one virtual network and three subnets, one for each tier.
